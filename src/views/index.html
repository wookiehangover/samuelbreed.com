Appendix

0.1 Deployment

Not to harp on the boring details, but I have some opinions about web publishing:

I did not use a complex tool to compose this website. (Deploying it, however, is a different story) I specifically chose to eschew the modern dev tool chains, blogging platforms, and web frameworks because I think that there's a simplicty of an html document that's easily forgotten and rarely appreciated. I'm not judging other techniques for publishing things on the web, but it's worth pointing out that the old ways still work.

Not having a build step or a line of JavaScript was beyond compare. But writing anything complex like this would be too hard. I scaled the barrier to enter this easy world of minimalist web authoring more than 20 years ago in a textarea on Geocities.

I have the pleasure of working with those types of tools regularly. Writing this in a single html file was a joy. Writing only 4 lines of CSS for this website was delightful.

But then I went and did a thing that neatly contradicts the sentiments above and deployed this "website" in a very modern way.

This website is deployed using a serverless function on a platform called Begin, which makes it easy to deploy applications to AWS Lambda. Which is just about the most complicated way to make an HTML file into a domain, but the deploys are faster than any tool I've ever used and they handle TLS certificate management so it was an easy decision.

0.2 Security

For the past few years I've spent a good deal of time at work thinking about browser security on a large website. From my experience, browser security can be summarized into a few concerns:

  1. Build & Publishing checks
    • Unless you're making a baroque statement about something or other, there will probably be a build process involved in building a website.
    • The outcome of any build process can either pass or fail. This is an opportunity to ensure that whatever build artifacts are produced contain up-to-date dependencies, pass lint and style checks, and pass any test suites.
  2. Runtime Restrictions
    • The best offense is a good defense. Treat the browser like an untrustable runtime: don't execute any code that you didn't author.
    • Content Security Policy is a somewhat blunt but very effective tool for eliminating what's allowed to run on your website.
    • The best 3rd party scripts are no 3rd party scripts.

As such, this website uses a Content Security Policy that's extremely restrictive. This is for everyone's benefit. I'm all for keeping whatever JavaScript is ferried along in browser extensions from interfering with this page's runtime, despite that there's nothing interesting going on.
